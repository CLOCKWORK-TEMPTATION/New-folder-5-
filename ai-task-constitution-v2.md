# دستور تنفيذ المهمة لمستودع البحث العميق متعدد الوكلاء — النسخة 2.1.0

---

## القسم صفر: التراتبية والسيادة

### 0.1 — غرض هذا الدستور

هذا الدستور يحكم أي عمل يخص هذا المستودع عندما يكون الهدف هو الوصول إلى:

- استدعاء تنفيذي موحد بين المنسق وباقي الوكلاء.
- تمرير حالة موحد وسليم بين مراحل البحث والاستخراج والتحليل والصياغة.
- جاهزية بناء وتشغيل كاملة لكل مشروع داخل المستودع.

هذا الدستور لا يصف الوضع المثالي نظرياً فقط، بل يربط بين **الحالة الحالية الفعلية** و**الحالة المستهدفة الملزمة**.

### 0.2 — الحقيقة المعمارية الملزمة لهذا المستودع

المنسق الرئيسي يعمل بإطار:

`AutoGen`

والوكلاء المنفذون يعملون بإطاري:

`deepagents`

`LangChain`

وعليه:

- يبقى المنسق هو نقطة القرار الوحيدة.
- يبقى كل وكيل منفذ مسؤولاً عن اختصاصه فقط.
- لا يجوز تغيير هذا الفصل المعماري إلا بطلب صريح وواضح من المستخدم.

### 0.3 — تراتبية البنود

عند تعارض بندين، يسود الأعلى:

| المستوى | النطاق | قابل لتجاوز المستخدم؟ |
|---|---|---|
| **0 — الصدق المعماري** | توصيف الواقع الحالي وعدم تزوير الجاهزية | لا |
| **1 — وحدة العقد** | شكل الاستدعاء ونتيجته وهوية المنسق والوكلاء | لا |
| **2 — سلامة الحالة** | الحفظ، التمرير، الأثر، تتبع الجولات | لا |
| **3 — جاهزية البناء والتشغيل** | البناء، التثبيت، الاعتماديات، أوامر التشغيل | نعم — إذا طلب المستخدم صراحةً تجاوزها مؤقتاً |
| **4 — التفضيلات الأسلوبية** | أسلوب التوثيق، ترتيب الأقسام، مستوى التفصيل | نعم |

### 0.4 — خط الأساس الفعلي بتاريخ 2026-03-12

الحالة الحالية لهذا المستودع ليست موحدة بعد، ويجب اعتبار البنود التالية حقائق تشغيلية لا يجوز تجاهلها:

- المنسق الحالي يستدعي الوكلاء عبر عمليات فرعية محلية، لا عبر خدمات شبكية مستقرة.
- الوكلاء الأربعة لا يشتركون حالياً في عقد إدخال واحد.
- بعض نقاط الدخول الحالية تستخدم بيانات تجريبية ثابتة بدلاً من مدخلات تشغيلية حقيقية.
- جاهزية البناء غير مكتملة في جميع المشاريع.

خلاصة البناء الحالية:

| المشروع | النتيجة الحالية |
|---|---|
| `search-manager-agent` | اجتاز فحص تجميع ملفات بايثون |
| `search-scout-agent` | فشل البناء بسبب اعتماديات وأنواع مفقودة |
| `content-extractor-agent` | فشل البناء بسبب مشاكل أنواع |
| `deep-research-analysis-agent` | اجتاز البناء |
| `report-drafting-agent` | فشل البناء لأن أداة البناء غير متاحة محلياً |

**قاعدة حاكمة:** ممنوع كتابة أي توثيق أو كود جديد يوحي بأن النظام "جاهز بالكامل" قبل إزالة هذه الفجوات فعلاً.

## القسم الأول: سير التنفيذ للمهمة داخل هذا المستودع

### 1.0 — تصنيف التعقيد

| المستوى | التعريف في هذا المستودع | المراحل المطلوبة |
|---|---|---|
| **بسيطة** | تعديل توثيقي أو إصلاح موضعي داخل وكيل واحد | 1.1 → 1.4 → 1.5 → 1.6 |
| **متوسطة** | توحيد نقطة دخول وكيل واحد أو إصلاح بناء مشروع واحد | 1.1 → 1.2 → 1.3 → 1.4 → 1.5 → 1.6 |
| **معقدة** | أي تغيير يمس المنسق مع أكثر من وكيل، أو بروتوكول الحالة، أو الجاهزية الكاملة للمستودع | جميع المراحل كاملة |

أي عمل يخص هذه المهمة الحالية يُصنّف **معقداً** افتراضياً.

### 1.1 — الاستيعاب

قبل أي تنفيذ يجب تحديد الآتي صراحةً:

- ما هو شكل الاستدعاء الحالي لكل مشروع.
- ما هو شكل الاستدعاء المستهدف الموحد.
- أين تبدأ الحالة المشتركة.
- أين تُحفظ.
- من يملك حق تعديلها.
- ما هي الفجوة الفعلية في البناء لكل مشروع.

**بوابة العبور:** القدرة على وصف الفرق بين الوضع الحالي والوضع المستهدف في جملة تشغيلية واحدة.

### 1.2 — التشخيص

أي خلل في هذا المستودع يجب تصنيفه ضمن واحد أو أكثر من الأنواع التالية:

- خلل عقد استدعاء.
- خلل تمرير حالة.
- خلل بناء أو اعتمادية.
- خلل في فصل المسؤوليات بين المنسق والوكلاء.
- خلل في التطابق بين التوثيق والسلوك الفعلي.

**بوابة العبور:** تحديد السبب الجذري، لا الاكتفاء بالأثر الظاهر.

### 1.3 — التخطيط

أي خطة تنفيذ لهذه المهمة تسير بالترتيب التالي:

1. تثبيت العقد التنفيذي الموحد.
2. تثبيت عقد الحالة المشتركة.
3. توحيد نقاط الدخول في الوكلاء الأربعة.
4. تكييف المنسق ليتحدث بالعقد الموحد فقط.
5. إصلاح البناء والاعتماديات لكل مشروع.
6. توثيق الفجوة المتبقية والتحقق النهائي.

ممنوع عكس هذا الترتيب إلا بسبب تقني موثق.

### 1.4 — التنفيذ

في التنفيذ الفعلي:

- يبدأ المنسق من طلب المستخدم.
- يصنع غلاف مهمة موحداً.
- يمرر الغلاف للوكيل المختص فقط.
- يستقبل نتيجة موحدة.
- يحدّث الحالة المشتركة.
- يقرر الخطوة التالية.

### 1.5 — التحقق الذاتي

لا يُعتبر العمل مكتملاً قبل التحقق من:

- وجود عقد إدخال موحد فعلاً.
- وجود عقد مخرجات موحد فعلاً.
- قابلية تتبع كل مرحلة عبر معرف تشغيل واحد.
- نجاح البناء لكل مشروع داخل نطاق المهمة.
- تطابق التوثيق مع السلوك الفعلي.

### 1.6 — التسليم

التسليم النهائي لهذه المهمة لا يكون وصفاً عاماً، بل مخرجاً قابلاً للتطبيق يتضمن:

- دستوراً واضحاً.
- تعريفاً للعقد الموحد.
- تعريفاً للحالة المشتركة.
- بوابات قبول جاهزية البناء.
- توثيقاً صريحاً للوضع الحالي والفجوة المتبقية إن وجدت.

## القسم الثاني: معايير الجودة الملزمة

### 2.1 — معايير إلزامية عامة

| # | المعيار |
|---|---|
| G1 | المنسق هو نقطة القرار الوحيدة بين المستخدم والوكلاء |
| G2 | كل وكيل يستقبل **غلاف مهمة موحداً واحداً** ويعيد **غلاف نتيجة موحداً واحداً** |
| G3 | لا يوجد مسار خامس خفي للاستدعاء خارج العقد الموثق |
| G4 | كل انتقال بين مرحلتين يمر عبر حالة مشتركة قابلة للتتبع |
| G5 | أي فجوة بين الواقع والتوثيق تُعلن صراحةً ولا تُخفى |

### 2.2 — معايير عقد الاستدعاء

القناة التنفيذية المرجعية لهذا المستودع هي **قناة محلية أولاً**:

- المنسق يستدعي الوكلاء محلياً.
- العقد نفسه يجب أن يظل صالحاً لاحقاً إذا تم نقله إلى خدمة شبكية.
- النقل المستقبلي للشكل الخارجي مسموح.
- تغيير شكل الحمولة الداخلية غير مسموح دون رفع نسخة البروتوكول.

**العقد التنفيذي الموحد الإلزامي**:

```json
{
  "protocolVersion": "research-task-envelope/v1",

  "parentTaskId": "optional-parent-task-id",
  "workflowStage": "search|extract|analyze|draft",
  "sender": "search-manager-agent",
  "targetAgent": "search-scout-agent|content-extractor-agent|deep-research-analysis-agent|report-drafting-agent",
  "objective": "original user objective",

    "outputFormat": "markdown|json|docx|pdf"

  },

    "timeoutSeconds": 300
  },

  }

```

**قواعده الملزمة**:

- لا يُرسل المنسق نصاً حراً بديلاً عن هذا الغلاف.
- لا يفسر الوكيل مدخلات من بيئته الداخلية كبديل عن هذا الغلاف.
- أي مدخل قديم يُحوَّل أولاً إلى هذا الغلاف عبر طبقة تكييف واضحة.

### 2.4 — معايير الحالة المشتركة

المصدر الوحيد المسموح للحالة المشتركة على مستوى الرحلة كاملة هو مجلد تشغيل مشترك على مستوى الجذر.

البنية المرجعية:

```text
runtime\
  runs\
    <run-id>\
      state\
        workflow-state.json
        research-context.json
      00-request\
        request-envelope.json
      01-search\
        input-envelope.json
        result-envelope.json
        search-results.json
      02-extraction\
        input-envelope.json
        result-envelope.json
        extracted-documents.json
      03-analysis\
        input-envelope.json
        result-envelope.json
        analysis-summary.json
        gaps.json
      04-report\
        input-envelope.json
        result-envelope.json
        final-report.md
      logs\
```

القواعد الملزمة:

- المنسق وحده يملك تحديث:

`workflow-state.json`

- كل وكيل يكتب داخل مجلد مرحلته فقط.
- ممنوع على أي وكيل تعديل ملفات مرحلة سابقة.
- التمرير بين المراحل يتم عبر **مراجع الأثر** داخل الحالة المشتركة، لا عبر نسخ النصوص الطويلة دون حاجة.
- إذا احتاج الوكيل بيانات كبيرة، يمرر المنسق مسار الأثر لا نصه الكامل.

### 2.5 — معايير فصل المسؤوليات

| الطرف | المسؤولية المسموح بها |
|---|---|
| المنسق | التخطيط، الترتيب، إعادة المحاولة، اتخاذ قرار جولة جديدة، إعلان الإكمال النهائي |
| وكيل البحث | جمع النتائج وترتيبها فقط |
| وكيل الاستخراج | جلب المحتوى وتطبيعه فقط |
| وكيل التحليل | تقييم الموثوقية واستخراج الحقائق والثغرات فقط |
| وكيل الصياغة | كتابة التقرير النهائي فقط |

**بند ملزم:** وجود ملف تقرير داخل وكيل التحليل لا يمنحه حق التسليم النهائي للمستخدم.

### 2.6 — معايير جاهزية البناء

لا يعد أي مشروع "جاهزاً" إلا إذا حقق جميع البنود التالية:

| # | المعيار |
|---|---|
| B1 | التثبيت ينجح من ملفات الاعتماديات الخاصة بالمشروع فقط |
| B2 | أمر فحص الأنواع أو ما يعادله متاح ويعمل |
| B3 | أمر البناء يعمل دون اعتماد على أدوات عالمية غير موثقة |
| B4 | أمر التشغيل الفعلي يقرأ العقد الموحد بدلاً من بيانات تجريبية |
| B5 | ملف البيئة المرجعي محدث ويغطي جميع المتغيرات المطلوبة |
| B6 | مسار التشغيل الإنتاجي لا يعتمد على أمثلة صلبة داخل الدالة الرئيسية |
| B7 | نتيجة التنفيذ تعيد رمز خروج صحيحاً ومخرجاً موحداً |

### 2.7 — معايير خاصة بهذا المستودع

- يجب أن يملك كل وكيل منفذ نقطة دخول موحدة لبيئة التطوير ونقطة دخول موحدة للإنتاج.
- يجب أن يشير:

`npm start`

إلى المخرج المبني، لا إلى المصدر الخام، في الوضع الإنتاجي.

- يجب إضافة:

`typecheck`

إلى كل مشروع من مشاريع

`TypeScript`

إذا كان غائباً.

- يجب أن يقبل المنسق غلاف المهمة أو أن يبنيه ثم يمرره دون تحويلات غير موثقة.

## القسم الثالث: المحظورات

### 3.1 — خطوط حمراء

- ممنوع أن يتخاطب وكيل منفذ مع وكيل منفذ آخر مباشرة.
- ممنوع أن يكون النص الحر بين المنسق والوكيل هو العقد المستقر.
- ممنوع استخدام بيانات مثال ثابتة في المسار الإنتاجي.
- ممنوع إعلان جاهزية بناء لمشروع لم يجتز أوامر التحقق فعلاً.
- ممنوع تغيير بنية الحالة المشتركة دون توثيق ورفع نسخة البروتوكول.

### 3.2 — محظورات تنفيذية

- منع إضافة مسار استدعاء جديد خاص بوكيل واحد إذا كان لا يخدم بقية الوكلاء.
- منع الاعتماد على:

`stdin`

في وكيل، وعلى وسيطات سطر الأوامر في وكيل ثان، وعلى بيانات صلبة في وكيل ثالث، كحل دائم.

- منع كتابة الآثار داخل مجلدات وقت تشغيل خاصة بكل وكيل إذا كانت الرحلة كاملة تحتاج تتبعاً مركزياً.
- منع الاعتماد على أدوات منصبة عالمياً مثل:

`tsc`

إذا لم تكن مثبتة ضمن المشروع أو مشغلة عبر مدير الحزم.

### 3.3 — محظورات تواصل وتوثيق

- ممنوع أن يقول التوثيق إن الاستدعاء شبكي إذا كان التنفيذ الفعلي محلياً فقط.
- ممنوع أن يقال إن العامل "مكتمل" إذا كان يحتاج طبقة تكييف إضافية غير منفذة.
- ممنوع استخدام أوصاف عامة مثل "موحد" أو "جاهز" دون معيار قابل للفحص.

### 3.4 — سياسة الافتراض

**ممنوع:** افتراض شكل الحالة أو شكل نتيجة الوكيل من دون الرجوع إلى النوع أو المخطط الفعلي.

**مسموح:** افتراض الممر التنفيذي المحلي كافتراض افتراضي لهذا المستودع، لأن هذا هو الواقع الحالي.

**القاعدة:**

1. إذا كان الغموض يغير شكل العقد أو ملكية الحالة أو معيار الجاهزية، يجب الحسم الصريح.
2. إذا كان الغموض تفصيلياً داخل نفس العقد، يُتخذ أفضل افتراض ويُعلن.
3. عند الشك، يحافظ التنفيذ على:

`AutoGen`

في المنسق وعلى:

`deepagents`

`LangChain`

داخل الوكلاء.

## القسم الرابع: الشفافية والمحاسبة

### 4.1 — إعلان العجز

إذا تعذر جعل أحد المشاريع جاهزاً:

1. يعلن اسم المشروع.
2. يعلن الأمر الذي فشل.
3. يعلن السبب النصي المباشر.
4. يعلن أثر الفشل على الرحلة الكاملة.

### 4.2 — بروتوكول فشل البناء

إذا فشل البناء في أي مشروع:

- لا تُمنح جاهزية كاملة للمستودع.
- تُسجَّل حالة المشروع في مصفوفة الجاهزية.
- لا يُوصَف العطل بأنه "تفصيلي" إذا كان يمنع الخروج المبني أو التشغيل القياسي.

### 4.3 — بروتوكول فشل الوكيل أثناء الرحلة

إذا أعاد الوكيل حالة:

`failed`

أو:

`partial`

أو:

`needs_more_research`

فإن المنسق فقط هو الذي:

- يقرر إعادة المحاولة.
- يزيد رقم الجولة.
- يحدث الحالة المشتركة.
- يقرر إعادة التوجيه أو الإنهاء.

### 4.4 — تطابق التوثيق مع الواقع

أي تغيير في أحد البنود التالية يجب أن ينعكس فوراً في الدستور والوثائق التشغيلية:

- شكل غلاف المهمة.
- شكل غلاف النتيجة.
- مجلد الحالة المشتركة.
- أوامر البناء.
- متغيرات البيئة.

`needs_more_research`

- يملأ حقل:

`handoff.recommendedNextStage`

بالقيمة:

`search`

- يضيف الاستعلامات المقترحة داخل:

`handoff.statePatch`

- يقرر المنسق وحده فتح الجولة التالية.

### 5.3 — إذا كانت المدخلات كبيرة جداً

- يمرر المنسق مسارات آثار بدلاً من النص الخام.
- يظل:

`workflow-state.json`

هو السجل الجامع.
- يمنع تقسيم الحقيقة التشغيلية بين رسائل محادثة متناثرة غير قابلة للاسترجاع.

### 5.4 — إذا احتفظ وكيل بذاكرة داخلية خاصة به

الذاكرة الداخلية مسموحة لتحسين الأداء أو الاستمرارية، لكنها لا تصبح المصدر المرجعي للرحلة الكاملة.

المصدر المرجعي الوحيد عبر جميع الوكلاء يظل:

`runtime\runs\<run-id>\state`

### 5.5 — إذا كان أحد الوكلاء غير جاهز بينما البقية جاهزة

- لا تُعلن الجاهزية الكاملة للمستودع.
- يمكن الإعلان عن جاهزية جزئية فقط مع تسمية المشاريع الجاهزة وغير الجاهزة.
- لا يُعتبر "وجود بديل يدوي" بديلاً عن اكتمال جاهزية المشروع نفسه.

### 5.6 — إذا كتب وكيل التحليل تقريراً داخلياً

يعامل هذا الملف كأثر تحليلي وسيط، وليس كتسليم نهائي للمستخدم.

التسليم النهائي يظل مسؤولية:

`report-drafting-agent`

بعد موافقة المنسق.

## القسم السادس: الحوكمة

### 6.1 — تعديل الدستور

أي تعديل على هذا الدستور يجب أن يتضمن:

1. ما الذي تغيّر.
2. لماذا تغيّر.
3. هل كسر التوافقية أم لا.
4. تاريخ التعديل.
5. هل تطلب رفع نسخة:

`protocolVersion`

أم لا.

### 6.2 — سياسة النسخ

النسخ الدلالية لهذا الدستور والبروتوكول تكون كالتالي:

- زيادة طفيفة إذا أضيف حقل اختياري أو توضيح غير كاسر.
- زيادة رئيسية إذا تغير حقل إلزامي أو معنى حالة أو بنية مجلد الحالة.

### 6.3 — تعريف الإكمال النهائي للمهمة الحالية

لا تعتبر المهمة الحالية منجزة بالكامل إلا عند تحقق جميع ما يلي معاً:

- المنسق يستدعي كل الوكلاء عبر عقد موحد واحد.
- الوكلاء الأربعة يقرؤون نفس شكل الغلاف.
- كل مرحلة تكتب أثرها في مجلد تشغيل مشترك واحد.
- كل مشروع يملك أوامر تثبيت وفحص وبناء وتشغيل عاملة.
- التوثيق يصف الواقع النهائي دون فجوة.
